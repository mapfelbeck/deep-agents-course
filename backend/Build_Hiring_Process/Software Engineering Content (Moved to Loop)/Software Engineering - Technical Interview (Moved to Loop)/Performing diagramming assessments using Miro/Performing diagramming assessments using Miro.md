# Performing diagramming assessments using Miro



| **Summary** | There are two types of diagram assessments that can be performed in an interview: Intepreting or understanding common diagrams and drawing diagrams. This page provides an overview of the tools used to create simple diagrams within an interview using Miro |
| --- | --- |
| **Audience** | * Interviewers * Candidates |
| **When** | In the Interview |

# Overview

There are two capbilities that could be assessed as related to diagrams in a technical interview:

* Can the candidate interpret and understand a diagram?
* Can the candidate create a diagram

# Can they interpret drawings?

In this scenario, we are going to ask the candidate to look at a drawing and ask them what it represents. This is an activity that is up to the interviewer to represent: the diagram could be a logical architecture, business logic flow, or infrastructure diagram: It just needs to be relevant or representative of the types of diagrams that a candidate will encounter on a day\-to\-day basis. This type of scenario is common for early career candidates as it is a common part of their day to day job to look at architectural drawings and have to implement them in code.

We will use the Miro board to present a diagram that is well understood by the interviewer and ask the candidate to describe what is happening in the diagram.

## Setup

* Identify one or two diagrams that you will use to assess the candidate’s ability. These should be in line with the candidate’s role target.
* Library of diagrams can be found here: [https://miro.com/app/board/o9J\_lDytZbg\=/](https://miro.com/app/board/o9J_lDytZbg=/)

## In Interview

* Ask the candidate if they understand what is happening in the diagram
* Ask the candidate if they see opportunities for improvement with the system (or algorithm or feature) that is expressed by the diagram

# Can they draw?

In  this scenario, we are going to ask the candidate to review a user story and express a diagram or two that reflects how they would propose implementing that user story.

Alternatively, we could ask the candidate to draw a common architecture (such as n\-tier or microservices) or to draw the architecture of a system that they have worked on.

## Setup

* If you are going to use a pre\-defined story, be sure that you have the story well understood and your expectations are understood so that you can assess the candidate’s ability to visualize it.

## In Interview

* Paste the text of the story into the whiteboard to use as a talking point.
* Ask the candidate to draw an appropriate diagram to represent to work of the story. Type of diagram is not particular: We do not prescribe diagramming methodology. Instead look to be sure that the candidate is creating something that you understand
