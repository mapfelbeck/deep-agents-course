# Sample Initial Interview Summary Email 1

Begin with one sentence summarizing whether we should or should not proceed with a candidate, followed by a brief explanation why.  If you are recommending we proceed with a candidate, point out any areas you think the in\-person interview should focus on.

Follow with an area\-by\-area breakdown of how the candidate did, including a rating on the scale from **Not Covered** to **Excellent** described at Software Engineering \- Initial Interview.

Here a sample email for a candidate for which you recommend we proceed. Note this is for a candidate interviewing for the Engineer role.

Reminder to not include any candidate PII (Personally Identifiable Information \- that could be used to identify, contact or locate an individual) in your feedback. [Link to PII training](https://rise.articulate.com/share/ilwziedjqh98CncqQydOlfTqiaQHt63w#/)

**Summary:** I strongly recommend we proceed with the candidate as a \[role, i.e. Engineer, Sr. Engineer, or SA]. This candidate mostly exceeded my expectations for an Engineer in their fundamentals and problem solving skills, and did well everywhere else. I think they'll be able to add velocity to a project right away and I'm excited at the thought of working with them!

* Technically, a very solid candidate.  Was knowledgeable in all categories.
* Minor reservations about their communication/interactivity.  We need to get a better feel for their teamwork and communication during the in\-person interview.

**Scale**: Not Covered / Poor / Fair / Good / Excellent

**Assessed as:** Engineer \[i.e. this is the role that your write up is for; i.e. this is where you state what role the scores below are assessed against]

**Communication – Fair/Good**

* Initially, very terse communication until I got the candidate talking about robotics.   Their interest in robotics drove them into technology.  


	+ Has created their own robot and won several competitions!
* Used a concrete example to explain most concepts, and did a good job doing so.

**Fundamentals – Good/Excellent**

* Solid understanding of hash table


	+ Knew lookup was constant time
	+ Discussed worst case scenario and knew it would be O(m) where m \= number of duplicate keys/hashes
* Interface – method signatures without the implementation


	+ Can implement as many interfaces as desired
* Abstract – method signatures and implementation


	+ Suggested that C\# and Java offer single inheritance
	+ When prompted, knew that C\+\+ provided multiple inheritance
* Generics


	+ Used a concrete example to explain them
	+ Said that generics can prevent code duplication
	+ List, Dictionary are types that use generics in C\#
* Callbacks


	+ Instruct a function to execute another function when it’s done
	+ Delegates and Lambdas can be used for callbacks in C\#
* Recursion vs Iteration


	+ Understood and explained both
	+ Mentioned that a frame is added on the stack for each level of recursion and knew that it leads to stack overflow

**Web – Good**

* Suggested jQuery for controls
* Suggested RegEx to validate user input  (a phone number in our scenario)
* + Had an understanding of the format of RegEx.  Hadn’t used recently
* Real\-time client\-side validation provides better user experience
* Server\-side validation would be a less optimal user experience and more load on the server
* Suggested CSS styling so that features like the background color of a website can be managed from a single place

**Database – Good**

* Knew the SQL statements for WHERE, COUNT, GROUP BY
* Got the DISTINCT keyword
* Limited familiarity with NoSQL databases, but understands relational well

**GenAI \- Good**

* Could describe prompt engineering
* Knew when to use RAG vs fine tuning

**Troubleshooting – Fair/Good**

* Knew 80 for HTTP and 443 for HTTPS
* Didn’t know port number for SSH


	+ Suggested that the FTP port was 21 (correctly) but didn’t know SSH was 20
* Suggested running a background app as a Windows service, accessed via service.msc


	+ Suggested event log for resolving issues with failed service. Had looked into it before
* Suggested SSH for connecting to remote boxes
* Was unfamiliar with RDP
* Suggested ping for diagnosing network connectivity
