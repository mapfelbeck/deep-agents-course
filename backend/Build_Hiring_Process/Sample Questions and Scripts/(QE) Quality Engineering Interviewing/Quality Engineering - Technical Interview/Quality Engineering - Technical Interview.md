The Skills Interview can be one of the most difficult screens to conduct, due to the breadth of content that is covered across software engineering and domain\-specific concepts and tools. In the span of 60\-75 minutes, interviewer must cover the QE candidate’s technical breadth, technical depth, and demonstration of problem\-solving skills. Technical interviewers must use their time as efficiently as possible in order to appropriately assess the candidate’s skills.

## Preparing for the Candidate

Review the feedback from the initial interview and the candidate’s resume. Both of these will be provided for you by the TA team and included in the calendar invite. Scan the written feedback for observations to explore further, since the initial screener is unlikely to have gone into them in depth.

Consider the candidate’s proposed level: what would be expected from somebody in that role? The Build Career Navigator can help provide guidance about the relative expectations for the candidate’s proposed leveling. It’s also good to think about a few Build QE team members that you know at that level, and what your expectations would be.

## Structure of Skills Interview

Within a technical skills interview, the interviewer must assess the candidate’s software engineering and software quality\-related skills. The areas of emphasis during the interview will depend on the candidate, but should always ensure coverage of the candidate’s knowledge of computer science and software engineering fundamentals. Ultimately, a successful technical interview will validate where the candidate’s skills are strong, as well as find the boundaries or limits of their technical knowledge and experience. 

An example of how interviews might focus on different areas:



| **Engineer** | **Senior Engineer** | **Architect** | **Senior Architect** |
| --- | --- | --- | --- |
| * Whiteboard Coding * Good / Bad Coding Practices * Testing an Everyday Object * Fundamentals of Agile | * Whiteboard Coding * Automation Framework Knowledge * Specialized Tool Knowledge * Quality in Agile | * Automation Framework Design * Leading Others * Tool / Framework Selection * Common Quality Problems in Agile | * Test Strategy Design * Automation Architecture * Complex Systems * Influencing Stakeholders |

Adjust the amount of time you are spending on any particular question based on how the candidate is performing. For example, if the resume, target level, and initial response to the question leads you to believe that the candidate is a strong coder, you probably don’t need to spend 30 minutes on a whiteboard coding question. It is perfectly acceptable to explain to the candidate up\-front that the interview will cover a breadth of topics, and that you may need to cut a question short in order to move on to a different area. 

One (final) note: some candidates may have technical skills that don’t quite fit the generic profile of a quality engineer. One example of this is the early career candidate that has little to no experience in software quality (software quality techniques are rarely taught within universities). Another example is the experienced candidate with a technical background (e.g. CS degree), but is in a job where they are not writing much, if any code (e.g. the “rusty” coder). If needed, ask for help from an experienced technical interviewer on how to structure the assessment for these candidates. 

## Technical Question Examples

The below questions are *examples* of the types of questions that we commonly ask. They can be tailored appropriately to different candidate levels, or used as the basis for a different question. Keep in mind that candidates can (and do) post interview questions to places like Glassdoor, so make sure that you modify your questions appropriately or come up with new ones. 

When asking a whiteboard coding question, make sure to avoid questions that are too simple or require some sort of “trick” or specialized knowledge in order to solve. A good resource for these types of questions is ***Cracking the Coding Interview***, by Gayle Laakman McDowell ([link](https://www.crackingthecodinginterview.com)). 



| Write a function to zero out rows / cols in a 2D arraySoftware EngineeringWrite a function that takes a two\-dimensional array (or array of arrays) of ints and returns a reference to an array that has the same data, but with all rows and columns where you found a “0” zero’d out. **Why this question**: This is just one of many “whiteboard” style programming questions.  It should evaluate the candidates ability to reason through algorithms. **Looking for:**We are not looking for perfect syntax knowledge in any language.  Pseudo code should be sufficient for this question.We are looking for the candidate’s ability to approach a programming problem and construct a solution.  The manner in which they deconstruct and approach the problem, and how they behave when stuck, is as important as getting a correct answer. **Red Flags**: Candidates that struggle with the basic concepts around programming and traversing / manipulating a simple data structure. Overly dependent on out\-of\-the\-box data structures and functions. Doesn’t pick up on reasonable hints. Starts coding without talking through the problem, or doesn’t talk at all. | **Expert Answer:*** Asks design\-level questions (can I allocate extra memory?  Should I validate input?) * Diagrams possible solutions before writing any code * Lists test cases (better: lists test cases before coding) * Thinks about optimization in multiple dimensions: space, time, readability, etc. * Runs through algorithm after finishing, validates correctness * Can optimize to be in\-place * Understands language specific implications of 2D arrays, or array\-of\-arrays if not supported  **Advanced Answer:*** Asks some clarifying questions and does some algorithm design before coding * Implements correct solution * Able to work through issues or optimize with suggestions * Creates some tests and runs through implementation to catch defects  **Base Answer:*** Should be able to understand the problem at a conceptual level and ask clarifying questions * Implements a workable approach without optimization * Should test algorithm with provided data |

| How do you test an API?Quality Engineering, Software Architecture, Test Tools**Question:** Assume you are responsible for testing a set of APIs that your team is developing. How would you approach this?**Why this question**: Software systems are frequently complex aggregations of independent APIs and applications. Understanding how to test applications at the API level is a critical component to an overall test strategy, even when the functionality is directly exposed via end\-user interface. **Looking for:**Curiosity and understanding of the overall system architecture, technologies used, etc. (the interviewer will need to have a design in mind). Knowledge of the different types of APIs and how they impact testing and testability.  Knowledge of common tools used to test APIs and their relative benefits and drawbacks.  Knowledge of the automation implications of testing at an API versus a UI.**Red Flags:** While earlier career candidates may not have experience with API testing, Architect / Senior Architect candidates must be reasonably proficient. Automation experience needs to be code\-centric; we do not endorse low\-code / no\-code test tool solutions. | **Expert Answer:*** Quickly seeks to understand overall architecture and solution design. * Understands and explains that "API" is a generic term, used to describe a diverse set of implementing technologies. * Knows how each of these impacts testing (REST vs SOAP vs binary vs queue messaging vs file based etc). * Experienced with the primary test tools used to test APIs (cURL, SoapUi, RestLet, Postman, etc). * Able to define a balanced test strategy and explain the pros / cons of automating at an API level * Strong experience with devops integration as part of a continuous deployment / continuous testing model.  **Advanced Answer:*** Candidate demonstrates broader knowledge of different types of APIs * Demonstrates expertise in a particular API implementation. * Can articulate an opinion on API test tools and the relative benefits of automation at an API level. * Understands how API testing fits into a balanced automation strategy; explains using concepts like the Test Automation Pyramid or E2E flow diagrams.  **Base Answer:*** Candidate can clearly explain their API testing experience with one or more tools * Can articulate some good practices and lessons learned * Provides a reasonable explanation of the value of API testing vs. UI testing. |

| 100% Automation?Quality Engineering, DevOps, Software Architecture**Question**: We all agree that higher automated test coverage is better.  Should we strive for 100% automated test coverage?**Why this question**: Automation is an ongoing investment, and is not always worth the cost.  This question allows the candidate to show knowledge of the benefits and drawbacks of automation as a tool for software quality. **Looking for:**An experienced candidate should have no trouble with this question.  They should be able to list types of tests that are more or less automatable.  They should be able to describe characteristics of software systems that make them more or less automatable.  The should be able to talk through the economics of automation, and all the variable that impact the ROI on an automation  **Red Flags**: Insistence on unrealistic or expensive goals. Lack of hands\-on experience with test automation (e.g. hands over tests to a “test engineer”). For Senior Engineer and above, endorsement of poor practices (e.g. automating complex E2E user actions, inverted pyramid, etc.) would be concerning. | **Expert Answer:*** Understanding that automation as an investment: what is the upfront capital cost?  What are expected maintenance costs?  How often will these tests be run, and for how long? * Gives examples of types of tests that are easy to automate: unit tests, API tests, etc, and hard to automate tests: E2E tests with shared resources, “look and feel” tests, etc. * Uses knowledge of Test Pyramid and related concepts to describe a balanced test automation strategy * Leads with whole team\-focused approach, devops integration, and reasonable metrics  **Advanced Answer:*** Disagrees with statement and can use Test Pyramid as support for argument * Can give examples of hard to automate tests, or situations where automation would challenging (and not worth investment)  **Base Answer:*** Knows this is this not advisable, but has difficulty explaining why. * Can give general examples of things that you probably shouldn’t automate. |
